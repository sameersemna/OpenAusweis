# PHASE 2B: Browser Security Analysis

**Status:** Design Phase  
**Date:** May 10, 2026  
**Focus:** Threat modeling, origin validation, session isolation, logging policies

---

## Executive Summary

OpenAusweis browser authentication presents a layered security model:
- **Defense in Depth:** Origin validation at extension, native host, and daemon layers
- **Least Privilege:** Sessions are isolated by UUID; only one active at a time
- **No Secrets in Logs:** PIN, session tokens, cryptographic material never logged
- **Trusted Components:** Daemon is authoritative; browser is untrusted until origin-validated

**Critical Assumption:** Desktop app (Tauri) and daemon are trusted; browser extension and native host are security boundaries.

---

## Threat Analysis

### Threat 1: Malicious Website Spoofs Allowed Origin

**Attack:**
- Attacker site at `https://evil-attacker.com` sends START_SESSION claiming `relying_party: https://authentic-site.bund.de`
- If validation is weak, malicious site can authenticate as legitimate site

**Mitigations:**
1. **Extension Layer:** Validate actual window.location.origin in content script before forwarding
2. **Native Host Layer:** Re-validate relying_party origin against policy (defense in depth)
3. **Daemon Layer:** Trust native host has validated; log relying_party for audit

**Verdict:** ✅ **Mitigated**

**Residual Risk:** If attacker controls domain matching allowlist suffix (e.g., owns `sub.bund.de`), they can impersonate legitimate site using same suffix. **Mitigation:** Use exact-origin allowlists for critical sites, suffixes for testing/internal-only sites.

---

### Threat 2: Malicious Browser Extension Hijacks Native Messaging

**Attack:**
- Attacker installs fake extension ID in native messaging manifest `allowed_origins`
- Attacker extension can now send START_SESSION requests to native host

**Mitigations:**
1. **Manifest Restriction:** `allowed_origins` in manifest strictly limits which extension IDs can connect
2. **Chromium:** Extension ID generated by store; only legitimate extension gets the ID
3. **Firefox:** Add-on ID in manifest; only official add-on signed by Mozilla has it

**Verdict:** ✅ **Mitigated by browser store controls**

**Residual Risk:** If attacker is store reviewer or has compromised build, they could publish malicious extension with valid ID. **Mitigation:** Use code signing, security reviews, publish only from verified accounts.

---

### Threat 3: Man-in-the-Middle on Daemon Socket

**Attack:**
- Attacker process tries to connect to daemon socket and impersonate extension

**Mitigations:**
1. **File Permissions:** Daemon socket is mode 0o600 (owner read/write only)
2. **XDG_RUNTIME_DIR:** Socket is in `/run/user/1000/`, owned by user 1000
3. **Unix Domain Socket:** No network exposure; only local processes can connect

**Verdict:** ✅ **Fully mitigated**

**Assumption:** No privilege escalation (attacker is not root).

---

### Threat 4: Replay Attack on Session

**Attack:**
- Attacker captures session_id and reuses it later (e.g., re-submit same PIN)

**Mitigations:**
1. **Session TTL:** Session expires 5 minutes after creation; UUID is unique per session
2. **State Machine:** Session progresses PinEntry → CardInteraction → Completed; can't re-enter PinEntry
3. **One-Time Use:** Once completed/failed, session is removed

**Verdict:** ✅ **Mitigated**

**Residual Risk:** If attacker captures session_id while in CardInteraction state, they can't re-use it (auth already committed). If captured during PinEntry, they can only submit PIN if they also have the card (not feasible without physical access).

---

### Threat 5: Session Fixation

**Attack:**
- Attacker predicts/guesses session_id and tries to monitor/hijack it

**Mitigations:**
1. **UUID v4:** Session_id is random UUID generated by daemon; cryptographically strong entropy
2. **No Persistence:** Session_id is not used for authentication; it's just state tracking
3. **No Session Tokens Sent to Browser:** Browser only knows session completed (boolean), not the session_id (except for logs)

**Verdict:** ✅ **Mitigated**

---

### Threat 6: TOCTOU on Session State

**Attack (Time-of-Check-to-Time-of-Use):**
- Browser checks session state at T1: state=PinEntry
- Daemon transitions state at T2: state=CardInteraction
- Browser thinks it's still PinEntry, submits PIN again

**Mitigations:**
1. **Watch Stream:** Extension doesn't poll state; it gets pushed updates
2. **Atomic Transitions:** Daemon transitions state and broadcasts update in single operation
3. **State Machine Enforcer:** SessionManager::submit_pin() rejects if state != PinEntry

**Verdict:** ✅ **Mitigated**

---

### Threat 7: Privilege Escalation via IPC

**Attack:**
- Browser extension sends malformed IPC message to trigger buffer overflow or similar in native host or daemon

**Mitigations:**
1. **Message Validation:** All requests validated against schema before processing
2. **Type Checking:** Rust's strong typing prevents buffer overflows
3. **Bounds Checking:** String lengths checked (handoff_id < 128 chars, pin < 64 chars)
4. **No Exec:** Native host never evals or execs user input

**Verdict:** ✅ **Mitigated**

---

### Threat 8: Denial of Service (Session Exhaustion)

**Attack:**
- Browser extension opens MAX_ACTIVE_SESSION_WAITS=32 sessions simultaneously to exhaust daemon resources

**Mitigations:**
1. **One Session Limit:** Daemon enforces at most 1 active session; second START_SESSION fails with SESSION_ALREADY_ACTIVE
2. **Session TTL:** Sessions auto-expire after 5 minutes
3. **Resource Limits:** Even if 32 sessions were allowed, daemon has finite memory; OS process limits apply

**Verdict:** ✅ **Mitigated**

---

### Threat 9: Information Leakage via Logs

**Attack:**
- PIN or session tokens are logged; attacker reads logs and replays

**Mitigations:**
1. **Logging Policy:** Never log PIN, session_id used as secret, or cryptographic material
2. **Code Review:** All tracing statements reviewed; PII filtered
3. **Structured Logging:** Use field-based logging; PIN field explicitly excluded

**Verdict:** ✅ **Mitigated**

**Verification:** Grep for "pin", "password", "secret" in background.js, main.rs, native-host/src/main.rs; ensure no unintended logs.

## Origin Validation Deep Dive

### Layer 1: Extension (Defense in Depth)

**File:** [apps/browser-extension/src/background.js]

**Function:** `isAllowedOrigin(origin, policy)`

**Logic:**
1. Check exact origins: `policy.exact.includes(origin)`
2. Check suffix matches: `policy.suffixes.some(suffix => origin.endsWith(suffix))`
3. Return true if either matches, else false

**Policy Source:**
- Load from `chrome.storage.local`:
  - `allowedExactOrigins`: `["http://localhost", "https://localhost"]`
  - `allowedSuffixes`: `[".bundid.de", ".bund.de"]`
- Defaults: Same as above
- User-configured: Via desktop app settings (future feature)

**Enforcement:**
- On content script START_SESSION: call `isAllowedOrigin(relying_party, policy)`
- If false: return error to content script, do NOT forward to native host
- If true: create request envelope, forward to native host

**Risk:** If `chrome.storage.local` is corrupted or cleared, defaults are used (which are permissive but safe for testing).

---

### Layer 2: Native Host (Primary)

**File:** [apps/native-host/src/main.rs]

**Function:** `load_origin_policy() → OriginPolicy`

**Policy Bundle Structure:**
```
~/.config/openausweis/origin-policy/
├── current → policies/policy-1715377800/ (symlink)
├── policies/
│   ├── policy-1715377800/
│   │   ├── policy.json
│   │   └── policy.sha256
│   └── policy-1715384000/
│       ├── policy.json
│       └── policy.sha256
```

**Policy Loading Steps:**
1. Check env var `OPENAUSWEIS_POLICY_DIR` (override root)
2. If not set, use `~/.config/openausweis/origin-policy/`
3. Read symlink `current/` to get versioned directory
4. Load `current/policy.json` (JSON with `allowed_exact_origins` and `allowed_suffixes`)
5. Load `current/policy.sha256` (one-liner: `<sha256-hex> policy.json`)
6. Verify: `sha256sum policy.json == <sha256-hex>`
7. If checksum fails, log error and fall back to defaults
8. If policy.json missing, use defaults

**Defaults:**
```rust
const DEFAULT_ALLOWED_EXACT_ORIGINS: &[&str] = &["http://localhost", "https://localhost"];
const DEFAULT_ALLOWED_SUFFIXES: &[&str] = &[".bundid.de", ".bund.de"];
```

**Environment Variable Overrides:**
- `OPENAUSWEIS_ALLOWED_EXACT_ORIGINS="http://localhost,https://localhost,https://test.example.com"`
- `OPENAUSWEIS_ALLOWED_SUFFIXES=".bundid.de,.bund.de,.example.com"`
- If set, used instead of policy bundle (useful for testing)

**Enforcement in START_SESSION:**
- Extract `relying_party` origin from request
- Call `is_allowed_origin(relying_party, policy)`
- If false: return error code `RP_NOT_ALLOWED` with message
- If true: forward START_SESSION to daemon

**Risk:** If environment variables are set, policy bundle is ignored (useful for override, but could be a misconfiguration trap).

---

### Layer 3: Daemon (Trusts Native Host)

**File:** [apps/daemon/src/main.rs]

**Validation:**
- Daemon does NOT re-validate origin
- Daemon trusts native host has already validated
- Daemon stores relying_party in session for audit/logging
- Daemon logs session_id + relying_party on creation

**Risk:** If native host is compromised, daemon will process invalid origins. **Mitigation:** Native host binary integrity should be verified during installation (e.g., via package signature).

## Session Isolation & Lifecycle Security

### Isolation Guarantees

**Invariant 1:** At most one active session per browser instance.
- Enforced by daemon: `if !guard.is_empty() { return Err(SessionAlreadyActive) }`
- If two tabs in same browser try to authenticate simultaneously, second fails

**Invariant 2:** Sessions are identified by UUID, not by browser identity.
- Multiple browser instances can be active simultaneously (multiple tabs in different windows)
- Each gets its own session_id

**Invariant 3:** Session lifecycle is immutable (no state rewinds).
- PinEntry → CardInteraction (one-way)
- CardInteraction → Completed or Error (terminal)
- No state transitions backward

**Invariant 4:** Session expires after TTL (5 minutes).
- If user doesn't complete authentication, session is automatically removed
- Prevents accumulation of stale sessions

---

## Cryptographic Material Handling

### What Is NOT Stored Locally

- **PIN:** Never stored; only transmitted in SUBMIT_PIN request to daemon
- **Session Tokens:** No tokens issued; sessions are tracked by UUID only
- **Private Keys:** Not generated locally; official AusweisApp2 bridge handles them
- **Card Secrets:** Card-bound; never transmitted over IPC

### What IS Stored

- **Policy Bundle:** JSON file with allowed origins (not cryptographic)
- **Session Metadata:** session_id, state, relying_party (not secret)
- **Metrics:** Request counts, error counts (not secret)

## Logging Policy

### Extension Background Script
- **Allowed:** `info: "Session started: ${session_id}", "origin: ${origin}", "state: ${state}"`
- **Forbidden:** `"pin: ${pin}"`, `"session_token: ${token}"`
- Level: `console.log/warn/error` for development; structured logs for production

### Native Host
- **Allowed:** `info: "origin_validated: ${origin}, allowed=true/false"`, `"forwarded START_SESSION to daemon"`
- **Forbidden:** `"pin: ${pin}"`, response payloads with secrets
- Level: `tracing::info/warn/error` to stderr

### Daemon
- **Allowed:** `info: "session_id=${uuid}, state=${state}, relying_party=${origin}"`
- **Forbidden:** `"pin: ${pin}"`, `"card_secret: ${secret}"`
- Level: `tracing::info/warn/error`

### Desktop App
- **Allowed:** `"session active"`, `"error: ${error_message}"`
- **Forbidden:** Same secrets as daemon
- Level: Structured Tauri logs

## Verification Procedures

### Code Review Checklist

- [ ] All `fetch_status()` calls in extension return only: healthy, readers, session state (no PIN)
- [ ] All `send(requestEnvelope)` calls in extension include request_id for tracing
- [ ] All daemon responses include error code and message (no raw Rust Debug output)
- [ ] No PIN variable printed in any tracing/console statement
- [ ] Policy bundle checksum verified on load (sha256sum -c)
- [ ] Session UUID is always UUID v4 (cryptographic randomness)
- [ ] Socket file permissions are 0o600 (read-write by owner only)

### Manual Security Audit

1. **Extension origin validation:**
   - Load extension, open DevTools
   - Visit `http://localhost:8080`, click login: verify message reaches background.js
   - Modify content script to send invalid origin (e.g., `https://evil.com`), verify rejected

2. **Native host policy validation:**
   - Start native host with valid policy bundle
   - Send START_SESSION with valid origin: verify forwarded to daemon
   - Send START_SESSION with invalid origin: verify rejected
   - Delete policy.sha256, verify fallback to defaults

3. **Session isolation:**
   - Open two browser windows, both visit localhost:8080
   - First window: click login, monitor daemon for session_id
   - Second window: click login, verify daemon rejects (SESSION_ALREADY_ACTIVE)
   - Cancel first session, second window: click login again, verify succeeds

4. **Logging audit:**
   - Enable all log levels (RUST_LOG=debug)
   - Run authentication flow
   - Grep logs for "pin", "password", "secret", "card": verify no matches

## Secure Development Practices

### Secret Management
- No secrets committed to git
- `.env.local` and `credentials.json` in `.gitignore`
- Use environment variables for policy overrides (no hardcoding)

### Dependency Scanning
- `npm audit` for JS dependencies (browser-extension, desktop)
- `cargo audit` for Rust dependencies (daemon, native-host)
- CI: fail on audit warnings

### Supply Chain
- Native messaging host binary: signed with gpg (future)
- Policy bundle: signed with project key (future)
- Snap/Flatpak: signed by store (inherent security model)

## Incident Response Plan

### If Native Host is Compromised

**Detection:**
- Unexpected origins being validated as allowed
- Logs show RP_NOT_ALLOWED for valid origins

**Response:**
1. Stop native host immediately: `killall openausweis-native-host`
2. Revoke/rotate policy bundle: update `current` symlink to last-known-good policy
3. Check for unauthorized policy bundles: `ls -la ~/.config/openausweis/origin-policy/policies/`
4. Reinstall native host from trusted source: `sudo apt install --reinstall openausweis-native-host`
5. Restart: `./scripts/run-native-host.sh`
6. Audit logs for affected sessions: grep for origin_not_allowed events

### If Daemon is Compromised

**Detection:**
- Sessions unexpectedly complete without PIN input
- Errors suggest unauthorized authentication

**Response:**
1. Kill daemon: `killall openausweis-daemon`
2. Check for unauthorized changes: `git status`, `git log --oneline -5`
3. Reinstall daemon: `sudo apt install --reinstall openausweis-daemon`
4. Restart: `./scripts/run-daemon.sh`
5. Review session history: `ls ~/.local/share/openausweis/session-history/`

### If Browser Extension is Compromised

**Detection:**
- Extension sends requests to unauthorized relying parties
- Users report unexpected authentication prompts

**Response:**
1. Disable extension: `chrome://extensions` → toggle off
2. Check manifest: verify `allowed_origins` is correct
3. Reinstall from store: uninstall, then install from Chrome Web Store / Mozilla Add-ons
4. Audit browser logs for suspicious requests

## Security Roadmap

**PHASE 2B:** Focus on origin validation hardening and session isolation (current).

**PHASE 3:**
- Add policy bundle signatures (asymmetric crypto for integrity)
- Add native host binary signature verification
- Add session signing (HMAC session_id to prevent replay)
- Add audit logging (immutable session history)

**Future:**
- Hardware-backed session keys (TPM on supported systems)
- Browser native session tokens (if browsers provide secure storage)
- Certificate pinning for relying parties (TLS+PKI)
