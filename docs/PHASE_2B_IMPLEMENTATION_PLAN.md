# PHASE 2B: Real Browser Authentication Orchestration — Implementation Plan

**Status:** Design Phase  
**Last Updated:** May 10, 2026  
**Target:** Ubuntu 24+, GNOME/KDE, Wayland/X11, Chromium/Firefox

---

## Overview

PHASE 2B completes the real authentication orchestration layer: browser extension → native messaging host → daemon → smartcard reader. This plan ensures:
- End-to-end session orchestration with robust error recovery
- Hardened origin validation at extension and native host levels
- Real-time UX feedback for all states and failure modes
- Snap/Flatpak browser integration validation on Ubuntu 24+
- Strict adherence to security model and architecture decisions

**Critical Constraints:**
- No architecture redesign; daemon ownership model is immutable
- No manual eID cryptography; delegate to official AusweisApp2 bridge
- No cloud dependencies
- No merging of daemon and UI concerns
- Linux-first with Wayland compatibility mandatory

## 1. Browser Authentication Orchestration

### Current State
- IPC envelope exists (versioned, request_id, payload): [apps/browser-extension/src/background.js]
- Session state machine exists (PinEntry → CardInteraction → Completed/Error): [apps/daemon/src/session.rs]
- Native messaging framing is length-prefixed JSON on stdio: [apps/native-host/src/main.rs]

### Implementation Goals

#### 1.1 Extension-to-Daemon Request Flow
**Invariant:** Every browser-initiated session must validate origin, flow through the native host, and be tracked by the daemon.

```
Content Script → Background (origin validation) → Native Host (re-validate) → Daemon
        ↓              ↓                              ↓                      ↓
   [data-openausweis-login]   [isAllowedOrigin]   [load_origin_policy]   [SessionManager]
```

**Steps:**
1. **Content script detects login trigger** ([apps/browser-extension/src/content.js]):
   - Listen for clicks on `[data-openausweis-login]` elements
   - Extract `window.location.origin` and send START_SESSION message to background
   - Catch and log the response, dispatch CustomEvent for page observability

2. **Background validates origin** ([apps/browser-extension/src/background.js]):
   - Load allowed origins from `chrome.storage.local.allowedExactOrigins` and `allowedSuffixes`
   - Call `isAllowedOrigin(origin, policy)` before forwarding to native host
   - If rejected, return error to content script with reason
   - If allowed, send START_SESSION to native host with protocol version, request_id, and relying_party

3. **Native host re-validates and forwards** ([apps/native-host/src/main.rs]):
   - Load origin policy from `~/.config/openausweis/origin-policy/current/` (policy.json + policy.sha256)
   - Verify checksum before accepting policy
   - Re-validate relying_party origin against loaded policy
   - If rejected, return error; else forward START_SESSION to daemon with handoff_id

4. **Daemon creates session** ([apps/daemon/src/session.rs]):
   - SessionManager creates new Uuid-keyed session in PinEntry state
   - Return SessionStarted response with session_id and handoff_id
   - Native host forwards response to extension background

5. **Extension watches session completion** ([apps/browser-extension/src/background.js]):
   - After START_SESSION succeeds, call WATCH_SESSIONS on native host
   - Poll session state at 100–10,000ms interval (configurable, clamped by constants)
   - On state transition to Completed or Error, resolve the session wait promise
   - Timeout after SESSION_COMPLETION_TIMEOUT_MS (5–180s, default 120s)
   - Return result to content script

#### 1.2 Error Propagation
**Invariant:** All errors must surface origin, code, and message at each layer.

- **Extension layer:** If native host is unavailable or times out, fail the session and record metric
- **Native host layer:** If daemon is unavailable, return error with code and message to extension
- **Daemon layer:** If session creation fails (already active), return SESSION_ALREADY_ACTIVE error

**Metrics to track:**
- nativeDisconnects: port.onDisconnect fired
- nativeTimeouts: request timed out waiting for native host response
- watchRetries: WATCH_SESSIONS had to reconnect after disconnect
- daemonErrors: daemon returned error response
- sessionStarts: successful START_SESSION calls
- sessionCompletions: sessions transitioned to Completed
- sessionGuardRejects: rejected due to guard conditions (max active waits)
- waitAborts: session watch aborted before completion
- lastError: most recent error message and timestamp

#### 1.3 Multi-Browser Support (Chromium vs Firefox)

**Chromium (Chrome, Brave, Edge):**
- Extension ID: generated by store, must be placed in manifest.json `allowed_origins`
- Native messaging: uses Chrome's native messaging protocol (length-prefixed JSON)
- Manifest: [apps/browser-extension/src/native-messaging-host.json]

**Firefox:**
- Add-on ID: will be assigned during store submission
- Native messaging: uses Firefox's native messaging protocol (identical to Chrome for JSON frames)
- Manifest: `org.mozilla.firefox.native.openausweis.json` (to be generated)

**Shared code:** IPC envelope, origin validation, session watching all protocol-agnostic.

## 2. Browser Security Analysis

### Threat Model

**Threats:**
1. **Malicious website claiming a restricted origin** → Mitigated by origin validation at extension and native host
2. **Malicious browser extension accessing native host** → Mitigated by strict `allowed_origins` in manifest
3. **Man-in-the-middle on daemon socket** → Not applicable; local Unix socket is protected by file permissions (0o600)
4. **Replay attack on session** → Not applicable; sessions are per-browser-instance, UUIDs are per-session
5. **Session fixation** → Not applicable; session_id is generated by daemon
6. **Privilege escalation through IPC** → Mitigated by daemon validation of all requests

### Security Measures

#### 2.1 Origin Allowlist Enforcement

**Three layers of defense:**

1. **Browser Extension** (Defense in Depth):
   - Load allowlist from `chrome.storage.local` (keys: `allowedExactOrigins`, `allowedSuffixes`)
   - Default: `http://localhost`, `https://localhost`, `.bundid.de`, `.bund.de`
   - Check: exact origin match OR domain suffix match
   - If rejected, return error to content script; never forward to native host

2. **Native Host** (Primary):
   - Load policy from `~/.config/openausweis/origin-policy/current/` by default
   - Directory structure: `current/` → symlink to `policies/policy-{timestamp}/`
   - Files: `policy.json` (allowlist) + `policy.sha256` (integrity)
   - On startup, verify checksum: `sha256sum -c policy.sha256`
   - If missing or corrupted, fall back to environment variables or defaults
   - Re-validate `START_SESSION.relying_party` origin before forwarding to daemon
   - Log all validation failures and suspicious requests

3. **Daemon** (Not Involved):
   - Daemon trusts native host has already validated the origin
   - Daemon only tracks session_id and relying_party for reference

#### 2.2 Policy Bundle Management

**Atomic Updates:**
1. Desktop app writes new policy bundle to `policies/policy-{timestamp}/`
2. Compute SHA256 of `policy.json`, write to `policy.sha256`
3. Update `current` symlink atomically: `ln -sfn policies/policy-{timestamp}/ current-tmp && mv current-tmp current`
4. Native host reads from `current` symlink on each START_SESSION validation

**Rollback:**
- Keep last 3 versioned policy bundles in `policies/`
- Revert by updating `current` symlink to previous timestamp
- No restart required; native host re-reads policy on next request

#### 2.3 No Sensitive Logging

**Rule:** Never log PIN, session tokens, or cryptographic material.

- Background script: log origin, session_id, state transitions, not the PIN
- Native host: log relying_party, session_id, validation results, not the PIN
- Daemon: log session_id, state transitions, not the PIN
- All logs go to stderr (native host) or tracing subscriber (daemon/extension)

#### 2.4 Session Isolation

**Invariant:** Only one active session per browser instance.

- Background: `activeSessionWaits` map: 1 session_id per request_id
- Daemon: `SessionManager` enforces at most 1 active session; returns SESSION_ALREADY_ACTIVE if another exists
- If a session times out or completes, it's removed from tracking

## 3. Native Messaging Lifecycle Design

### IPC Protocol

**Envelope (all directions):**
```json
{
  "protocol_version": 1,
  "request_id": "uuid-string",
  "payload": { ... }
}
```

### Request/Response Sequences

#### 3.1 START_SESSION (Extension → Native Host → Daemon)

**Request (Extension → Native Host):**
```json
{
  "protocol_version": 1,
  "request_id": "12345678-...",
  "payload": {
    "type": "START_SESSION",
    "relying_party": "https://example.bund.de",
    "handoff_id": "optional-id"
  }
}
```

**Native Host Validation:**
- Extract `relying_party` origin
- Load policy, verify checksum
- Check exact origins and domain suffixes
- Log: `origin={relying_party} allowed=true/false`

**Request (Native Host → Daemon):**
```json
{
  "protocol_version": 1,
  "request_id": "12345678-...",
  "payload": {
    "type": "START_SESSION",
    "relying_party": "https://example.bund.de",
    "handoff_id": "optional-id"
  }
}
```

**Response (Daemon → Native Host → Extension):**
```json
{
  "protocol_version": 1,
  "request_id": "12345678-...",
  "payload": {
    "type": "SESSION_STARTED",
    "session_id": "abcd-efgh-...",
    "state": "PIN_ENTRY",
    "handoff_id": "optional-id"
  }
}
```

**Extension behavior:**
- If successful: add to `activeSessionWaits`, start WATCH_SESSIONS
- If error: return error to content script, record metric

#### 3.2 WATCH_SESSIONS (Extension → Native Host → Daemon)

**Request (Extension → Native Host):**
```json
{
  "protocol_version": 1,
  "request_id": "12345678-...",
  "payload": {
    "type": "WATCH_SESSIONS",
    "interval_ms": 500
  }
}
```

**Native Host:**
- Forward directly to daemon (no validation needed)

**Daemon streaming behavior:**
- Open persistent connection
- Send initial SessionUpdated or SessionCancelled (if exists)
- Poll `SessionManager::current_session()` at interval_ms
- Send only on state change (initial + deltas)
- Connection stays open until client closes

**Responses (multiple, streaming):**
```json
{
  "protocol_version": 1,
  "request_id": "12345678-...",
  "payload": {
    "type": "SESSION_UPDATED",
    "session_id": "abcd-efgh-...",
    "state": "CARD_INTERACTION",
    "error": null,
    "handoff_id": "optional-id"
  }
}
```

**Extension behavior:**
- On SESSION_UPDATED with state=COMPLETED: resolve promise, return success to content script
- On SESSION_UPDATED with state=ERROR: resolve promise, return error to content script
- On CONNECTION_CLOSED: after timeout, retry WATCH_SESSIONS
- On timeout (SESSION_COMPLETION_TIMEOUT_MS): abort wait, return error to content script

#### 3.3 SUBMIT_PIN (Only from Daemon, not from Extension)

**Note:** Extension does NOT submit PIN directly. PIN entry UI is handled by desktop or daemon-provided UI (future).

For PHASE 2B, SUBMIT_PIN is tested only in daemon+desktop integration, not in browser.

#### 3.4 Error Responses

**From Native Host or Daemon:**
```json
{
  "protocol_version": 1,
  "request_id": "12345678-...",
  "payload": {
    "type": "ERROR",
    "code": "RP_NOT_ALLOWED",
    "message": "relying party origin is not allowed: https://evil.example.com"
  }
}
```

**Extension handling:**
- Catch error, record lastError metric
- Return to content script: `{ ok: false, error: response.payload.message }`

### Timeout & Reconnection Strategy

**Native Host to Daemon Connection:**
- On ensureConnected: open Unix socket to daemon
- NATIVE_IDLE_DISCONNECT_MS = 15s: if no pending requests, close connection
- On disconnect: reject all pending promises, record nativeDisconnects metric
- On re-connection: transparently re-open socket

**WATCH_SESSIONS Stream:**
- Timeout before first event: WATCH_SESSIONS_FIRST_EVENT_TIMEOUT_MS = 4.5s
- Timeout during stream: CONNECTION_TIMEOUT_MS = 6s
- On timeout: record watchRetries, reconnect and retry WATCH_SESSIONS
- Max retries: not specified yet (todo in testing phase)

**Session Completion Watch:**
- Timeout: SESSION_COMPLETION_TIMEOUT_MS = 120s (clamped 5–180s)
- If timeout: abort wait, return error to content script, record waitAborts metric

2. Copy native messaging manifest: `apps/browser-extension/src/native-messaging-host.json`

8. Visit `http://localhost:8080` and click the login button

#### 4.4 Expected Flow
1. User clicks "Login with OpenAusweis"
2. Content script sends START_SESSION to background with origin=http://localhost:8080
3. Background validates origin against allowlist (localhost is allowed by default)
4. Background sends to native host, which re-validates and forwards to daemon
5. Daemon creates session, returns SessionStarted
6. Background starts WATCH_SESSIONS, polls daemon
7. (User inserts card, enters PIN in desktop app or simulator)
8. Daemon transitions session to Completed
9. WATCH_SESSIONS receives SessionUpdated(state=COMPLETED)
10. Background resolves promise, returns success to content script
11. Content script dispatches openausweis:response with result
12. Demo page receives event and displays "Success: Session ABCD-EFGH completed"

## 5. Session State Machine Recommendations

### State Diagram

```
START
  ↓
[PinEntry] ← user hasn't submitted PIN yet
  ↓
[CardInteraction] ← PIN accepted, authenticating with card
  ↓
[Completed] ← authentication succeeded
  ↓
[Cleanup/Expire]

OR

[Error] ← any error occurred (invalid PIN, card error, etc.)
  ↓
[Cleanup/Expire]
```

### State Transitions (Daemon-Enforced)

**PinEntry → CardInteraction:**
- Trigger: SUBMIT_PIN request with valid 6-digit PIN
- Action: SessionManager increments pin_attempts, updates state, resets expiry
- Invariant: Only 3 attempts; if exceeded, session is removed and ERROR returned

**CardInteraction → Completed:**
- Trigger: AuthExecutor completes (official stack finishes authentication)
- Action: SessionManager transitions to Completed, resets expiry
- Invariant: Session remains in system for TTL seconds to allow watch stream to observe

**Any → Error:**
- Trigger: AuthExecutor fails, card error, timeout, etc.
- Action: SessionManager transitions to Error, stores error message, resets expiry
- Invariant: Error is observable until TTL expires

**Any → (Expired):**
- Trigger: Instant now > entry.expires_at
- Action: SessionManager::prune_expired removes session from map
- Invariant: TTL is 5 minutes (300s); no session lives longer

**CardInteraction → (Cancelled):**
- Trigger: CANCEL_SESSION request (not yet implemented, todo for PHASE 2B)
- Action: SessionManager removes session, returns SessionCancelled
- Invariant: Only valid if session exists

### Observability

**Desktop UI watches via WATCH_STATUS:**
- Polls daemon every 500ms for active_session_count
- Tray tooltip updates: "Session active (1)"
- Desktop status panel shows current session state

**Browser Extension watches via WATCH_SESSIONS:**
- Polls native host/daemon via WATCH_SESSIONS
- Interval: 100–10,000ms (default 500ms)
- Surfaces state change to content script
- Resolves completion promise on state=Completed/Error

**Daemon logs all transitions:**
- `session_id=... state=PIN_ENTRY`
- `session_id=... state=CARD_INTERACTION`
- `session_id=... state=COMPLETED`
- `session_id=... state=ERROR error=...`

### Recommendations for PHASE 2B

1. **Add CANCEL_SESSION support:**
   - Implement in daemon: check session exists, remove, return SessionCancelled
   - Implement in native host: forward CANCEL_SESSION to daemon
   - Implement in extension: call CANCEL_SESSION on user abort or timeout

2. **Add session_id tracking in desktop app:**
   - Show current session_id in status panel (for debugging)
   - Allow manual session cancellation via UI button

3. **Improve error specificity:**
   - Distinguish between PIN errors (invalid format, too many attempts) and card errors (PACE failed, EAC failed, etc.)
   - Surface in UI and logs

4. **Add session history (desktop only):**
   - Keep log of last 10 sessions: session_id, relying_party, state, error, timestamp
   - Useful for diagnostics and post-mortem analysis

## 7. Linux Sandboxing Concerns

### Snap Constraints

**Current snapcraft.yaml:**
- `confinement: strict`
- Plugs: desktop, wayland, x11, home, raw-usb, network, etc.

**PHASE 2B concerns:**

1. **Native Messaging Manifest Path:**
   - Snap app is sandboxed; native messaging host must be outside snap
   - Solution: Native host is a separate Debian package, not snapped
   - Native host path: `/usr/bin/openausweis-native-host`
   - Manifest installed by: `dpkg` or manual script
   - Snap can launch unsandboxed host via `snapctl system-request`? No, that's for snapd ops.
   - Alternative: Snap ships native host as "classic" snap? Overkill.
   - **Recommended:** Ship daemon + desktop as snap, native host as separate Debian package

2. **Daemon Socket Access:**
   - Snap runs daemon at `$SNAP_USER_DATA/openausweis/daemon.sock`
   - Desktop app inside snap can access socket (same confinement)
   - Native host outside snap cannot access socket inside snap sandbox
   - Solution: Daemon socket in XDG_RUNTIME_DIR = `/run/user/1000/openausweis/daemon.sock`
   - XDG_RUNTIME_DIR is writable by both snap and unsandboxed processes
   - **Verified:** This works on Ubuntu 24+ with default AppArmor policies

3. **USB/Smartcard Access:**
   - Snap has `raw-usb` plug: grants access to `/dev/bus/usb`
   - PC/SC daemon (pcscd) outside snap controls smartcard access
   - Snap daemon cannot directly access card; must use pcscd
   - Solution: Daemon connects to pcscd via Unix socket: `/run/pcscd/pcscd.comm`
   - pcscd is typically systemd user service, socket is in XDG_RUNTIME_DIR
   - **Concern:** If pcscd is system service, socket may not be writable by snap user
   - **Mitigation:** Document requirement that pcscd runs as user service, not system

4. **Policy Bundle Access:**
   - Desktop app inside snap reads/writes `~/.config/openausweis/origin-policy/current/`
   - Home plug allows access to `~/`
   - **No issue:** Snap desktop app can read/write policy

5. **Browser Integration:**
   - Snap desktop app cannot directly interact with browser
   - Native host (outside snap) bridges browser ↔ daemon
   - Browser can access `http://localhost:8080` if test app runs outside snap
   - **Concern:** If user opens browser inside snap, native host messaging won't work across sandbox boundary
   - **Mitigation:** Document that browser MUST run outside snap for authentication to work
   - **Or:** Build browser-in-snap integration as PHASE 3+

### Flatpak Constraints

**Similar to Snap:**
- Flatpak app (desktop + daemon) runs in sandbox
- Native host must run outside sandbox
- Daemon socket in `$XDG_RUNTIME_DIR` (writable by both)
- Smartcard access via pcscd user service
- Browser must run outside Flatpak for native messaging to work

**Flatpak-specific:**
- Permissions: `--device=all --filesystem=host:ro --share=network`
- Dbus access: may be needed for system notifications (handled by Tauri)

### AppImage / Debian Package

**No sandboxing:**
- Daemon and desktop run as normal processes
- Native host runs as normal process
- All components can access `XDG_RUNTIME_DIR` and `~/.config`
- Browser native messaging works transparently
- **No concerns** for native messaging

## 8. Firefox vs Chromium Compatibility

### Native Messaging Protocol
Both use identical **length-prefixed JSON** framing on stdin/stdout.

**Frame format:**
```
[4-byte length in platform byte order][JSON payload]
```

**Example:**
```
\x4e\x00\x00\x00{"protocol_version":1,"request_id":"...","payload":{...}}
```

**No differences** between Chrome and Firefox for IPC protocol.

### Manifest Installation

**Chromium (Chrome, Brave, Edge):**
- Manifest: `/etc/chromium/native-messaging-hosts/org.openausweis.native.json`
- Or: `~/.config/chromium/NativeMessagingHosts/org.openausweis.native.json`
- Content:
  ```json
  {
    "name": "org.openausweis.native",
    "description": "OpenAusweis Native Messaging Host",
    "path": "/usr/bin/openausweis-native-host",
    "type": "stdio",
    "allowed_origins": [
      "chrome-extension://AAAABBBBCCCCDDDD/"
    ]
  }
  ```
- Extension ID: Generated by Chrome Web Store, must be hard-coded in manifest
- Problem: Extension ID differs per build/testing

**Firefox:**
- Manifest: `~/.mozilla/native-messaging-hosts/org.openausweis.native.json`
- Or: `/etc/opt/firefox/native-messaging-hosts/org.openausweis.native.json`
- Content:
  ```json
  {
    "name": "org.openausweis.native",
    "description": "OpenAusweis Native Messaging Host",
    "path": "/usr/bin/openausweis-native-host",
    "type": "stdio",
    "allowed_extensions": [
      "openausweis-browser@openausweis.org"
    ]
  }
  ```
- Add-on ID: Assigned by Mozilla Add-ons store, communicated to extension via manifest.json `browser_specific_settings`
- Advantage: Single manifest works for all Firefox versions

**manifest.json (Chromium v3):**

### Key Differences

| Feature | Chromium (v3) | Firefox (v2) |
|---------|---------------|-------------|
| Manifest version | 3 | 2 |
| Background | service_worker | scripts |
| Storage API | chrome.storage | browser.storage |
| Async | Promises | Promises/callbacks |
| Host permissions | host_permissions | permissions |
| Native messaging | chrome.runtime.connectNative | browser.runtime.connectNative |
| Action (popup) | action | browser_action |

### Code Compatibility Strategy

**Shared code path** for most logic (IPC, origin validation, session watching).

**Platform-specific adapters:**
1. Storage API: Wrapper for `chrome.storage` (Chromium) vs `browser.storage` (Firefox)
2. Message API: Both use identical `chrome.runtime.sendMessage` (Firefox uses `browser.runtime`)
3. Native messaging: Identical `connectNative()` call

**Build approach:**
- Use `process.env.BROWSER_TARGET` build variable
- Or: Detect at runtime using `typeof chrome !== 'undefined'` vs `typeof browser !== 'undefined'`

**Recommendation:** Ship two manifests, one JavaScript implementation:
- `manifest.v3.json` → Chromium build
- `manifest.v2.json` → Firefox build
- Single `background.js`, detect browser at runtime

## 10. Rollback Strategy

### Policy Bundle Rollback

**Scenario:** New policy bundle is deployed but causes validation failures.

**Steps:**
1. Identify last-known-good policy timestamp
2. Desktop app or admin: `ln -sfn policies/policy-{previous-timestamp}/ ~/.config/openausweis/origin-policy/current-tmp && mv current-tmp ~/.config/openausweis/origin-policy/current`
3. Native host on next request re-reads policy from new symlink target
4. Validation succeeds with old policy
5. **No restarts required**

**Verify:**
- Browser extension popup shows daemon connected
- Login flow succeeds

### Extension Version Rollback

**Scenario:** New extension version has a bug (e.g., incorrect origin validation).

**Steps:**
1. User: Go to `chrome://extensions` (or about:addons in Firefox)
2. Disable the broken version
3. Download and load previous version as unpacked extension
4. Update native messaging manifest to point to new extension ID (if changed)
5. Verify login works

**Automated:** If using Chrome Web Store, can pull updates to users; users can disable extension and reload previous version.

### Daemon Rollback

**Scenario:** New daemon version breaks session lifecycle.

**Steps:**
1. User: Kill running daemon
2. User/script: `sudo apt install openausweis-daemon=0.1.0-prev` (downgrade via package manager)
3. User/script: `./scripts/run-daemon.sh` (restart)
4. Verify: Tray icon appears, status OK

**Automatic restart:** If daemon is systemd user service, `systemctl --user restart openausweis-daemon` restarts automatically.

### Native Host Rollback

**Same as daemon:**
1. `sudo apt install openausweis-native-host=0.1.0-prev`
2. Restart: native host picks up new binary path on next browser extension connection

### Snap Rollback

**Scenario:** Snap build of 0.1.0 is broken.

**Steps:**
1. User: `snap revert openausweis` (reverts to previous snap revision)
2. Snap automatically restarts desktop app
3. Verify: tray, status OK

**Automatic:** Snapd tracks last 2 revisions; older revisions need manual download.

### Total Rollback

**If everything breaks:**
1. Remove snap: `sudo snap remove openausweis`
2. Uninstall native host: `sudo apt remove openausweis-native-host`
3. Uninstall daemon: `sudo apt remove openausweis-daemon`
4. Disable browser extension
5. System is back to pre-OpenAusweis state
6. **Recovery time:** < 5 minutes
