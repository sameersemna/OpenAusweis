# PHASE 2B: Firefox vs Chromium Compatibility

**Status:** Design Phase  
**Date:** May 10, 2026  
**Scope:** Native messaging protocol, manifest differences, code compatibility

---

## Executive Summary

Firefox and Chromium-based browsers (Chrome, Brave, Edge) share the same **native messaging protocol** (length-prefixed JSON), but differ in:
1. Extension manifest format (v2 vs v3)
2. API naming conventions (`chrome.*` vs `browser.*`)
3. Manifest installation paths
4. Extension ID/add-on ID handling

**Goal:** Single JavaScript implementation with platform-specific manifests; runtime detection for API differences.

---

## Native Messaging Protocol (Shared)

### Frame Format

**Both Firefox and Chromium use identical framing:**

```
[4-byte length in platform byte order (little-endian on x86)][JSON payload]
```

**Example:**
```
Message: {"protocol_version":1,"request_id":"abc-123","payload":{"type":"START_SESSION","relying_party":"https://site.bund.de"}}
Length: 0x7b (123 bytes in decimal)
Bytes: 0x7b 0x00 0x00 0x00 {...json...}
```

**Advantage:** Single native host binary works with both browsers.

### Request/Response Protocol

**Identical between Firefox and Chromium:**

```json
{
  "protocol_version": 1,
  "request_id": "uuid-string",
  "payload": { ... }
}
```

**No differences in IPC semantics.**

## Manifest Differences

### Manifest v3 (Chromium: Chrome 88+, Edge, Brave)

**File:** `manifest.json` (v3 format)

```json
{
  "manifest_version": 3,
  "name": "OpenAusweis",
  "version": "0.1.0",
  "description": "German eID login for Linux",
  
  "background": {
    "service_worker": "background.js"
  },
  
  "content_scripts": [
    {
      "matches": ["*://*:*/*"],
      "js": ["content.js"]
    }
  ],
  
  "permissions": [
    "nativeMessaging",
    "storage"
  ],
  
  "host_permissions": [
    "*://*"
  ],
  
  "action": {
    "default_popup": "popup.html",
    "default_title": "OpenAusweis Status"
  },
  
  "native_messaging": [
    "org.openausweis.native"
  ]
}
```

**Key Features:**
- `service_worker` instead of `background.scripts` (Web Workers, not persistent background page)
- `native_messaging` array declares native host names
- `host_permissions` for broad origin access

### Manifest v2 (Firefox: All versions)

**File:** `manifest.json` (v2 format)

```json
{
  "manifest_version": 2,
  "name": "OpenAusweis",
  "version": "0.1.0",
  "description": "German eID login for Linux",
  
  "background": {
    "scripts": ["background.js"]
  },
  
  "content_scripts": [
    {
      "matches": ["*://*:*/*"],
      "js": ["content.js"]
    }
  ],
  
  "permissions": [
    "nativeMessaging",
    "storage",
    "http://*",
    "https://*"
  ],
  
  "browser_action": {
    "default_popup": "popup.html",
    "default_title": "OpenAusweis Status"
  },
  
  "browser_specific_settings": {
    "gecko": {
      "id": "openausweis-browser@openausweis.org",
      "strict_min_version": "109.0"
    }
  }
}
```

**Key Features:**
- `background.scripts` (persistent background page)
- `permissions` includes origin patterns (no separate `host_permissions`)
- `browser_action` instead of `action`
- `browser_specific_settings` for Firefox-specific metadata (add-on ID, min version)

## Extension ID Handling

### Chromium

**Extension ID:**
- Generated by Chrome Web Store during submission
- Fixed for published extension
- Example: `abcdefghijklmnopqrstuvwxyzabc`
- Format: 32 characters, lowercase alphanumeric

**Manifest Requirement:**
- Native messaging host manifest must include exact extension ID in `allowed_origins`
- File: `/etc/chromium/native-messaging-hosts/org.openausweis.native.json`

**Native Host Manifest (Chromium):**
```json
{
  "name": "org.openausweis.native",
  "description": "OpenAusweis Native Messaging Host",
  "path": "/usr/bin/openausweis-native-host",
  "type": "stdio",
  "allowed_origins": [
    "chrome-extension://abcdefghijklmnopqrstuvwxyzabc/"
  ]
}
```

**Multi-Browser Support:**
- If Brave or Edge versions are published, their extension IDs are different
- Separate manifest files needed, or `allowed_origins` includes all

---

### Firefox

**Add-on ID:**
- Assigned by Mozilla Add-ons (AMO) during submission
- Fixed for published add-on
- Example: `openausweis-browser@openausweis.org`
- Format: `{uuid-or-email}` as string

**Manifest Requirement:**
- Declared in extension manifest via `browser_specific_settings.gecko.id`
- Native messaging host manifest includes add-on ID in `allowed_extensions`
- File: `~/.mozilla/native-messaging-hosts/org.openausweis.native.json` (or system path)

**Native Host Manifest (Firefox):**
```json
{
  "name": "org.openausweis.native",
  "description": "OpenAusweis Native Messaging Host",
  "path": "/usr/bin/openausweis-native-host",
  "type": "stdio",
  "allowed_extensions": [
    "openausweis-browser@openausweis.org"
  ]
}
```

## API Differences

### Chrome API vs Browser API

| API | Chrome (Chromium v3) | Firefox (v2) |
|-----|----------------------|--------------|
| Storage | `chrome.storage` | `browser.storage` |
| Messages | `chrome.runtime.sendMessage` | `browser.runtime.sendMessage` |
| Native Messaging | `chrome.runtime.connectNative` | `browser.runtime.connectNative` |
| Tabs | `chrome.tabs` | `browser.tabs` |
| Permissions Listener | `chrome.runtime.onInstalled` | `browser.runtime.onInstalled` |

**Key:** Firefox uses `browser.*`, Chromium uses `chrome.*`; otherwise APIs are similar.

### Runtime Detection

**Strategy:** Detect browser at runtime; use appropriate API.

**Implementation:**
```javascript
// Detect browser API namespace
const API = typeof chrome !== 'undefined' ? chrome : (typeof browser !== 'undefined' ? browser : null);

if (!API) {
  throw new Error("Neither chrome nor browser API available");
}

// Use API for storage
API.storage.local.get(['key'], (result) => {
  console.log(result);
});

// Use API for native messaging
const port = API.runtime.connectNative("org.openausweis.native");
```

**Alternative:** Use polyfill library (e.g., `webextension-polyfill`) to normalize API.

## Shared Code Path

### background.js

**File:** `apps/browser-extension/src/background.js`

**Single implementation works for both:**
- Origin validation logic: browser-agnostic
- Native messaging: identical protocol (both use same connectNative call)
- Session watching: identical state machine
- Storage access: use `API.storage` (detected at runtime)

**Example:** Check both manifest files for differences (none in logic, only in metadata).

## Content Script Compatibility

### content.js

**File:** `apps/browser-extension/src/content.js`

**Shared implementation:**
```javascript
const API = typeof chrome !== 'undefined' ? chrome : browser;

const LOGIN_TRIGGER_SELECTOR = "[data-openausweis-login]";

function setupLoginBridge() {
  const trigger = document.querySelector(LOGIN_TRIGGER_SELECTOR);
  if (!trigger) {
    return;
  }

  trigger.addEventListener("click", async () => {
    const response = await API.runtime.sendMessage({
      type: "START_SESSION",
      relying_party: window.location.origin,
    });

    window.dispatchEvent(new CustomEvent("openausweis:response", { detail: response }));

    if (!response?.ok) {
      console.warn("OpenAusweis bridge error", response?.error);
      return;
    }
  });
}

setupLoginBridge();
```

**No browser-specific code needed.**

## Build Strategy

### Option A: Single Extension with Runtime Detection

**Pros:**
- One code base, easy to maintain
- Single extension ID for Chromium, single for Firefox

**Cons:**
- Runtime detection adds minor overhead
- Manifest files still differ

**Implementation:**
1. Create `manifest.v3.json` (Chromium)
2. Create `manifest.v2.json` (Firefox)
3. Use build script to select correct manifest during packaging
4. Single `background.js`, `content.js`, `popup.js` for both

**Build Command (example):**
```bash
# Build for Chromium
cp manifest.v3.json manifest.json && npm run build:chromium

# Build for Firefox
cp manifest.v2.json manifest.json && npm run build:firefox
```

### Option B: Separate Extension Codebases

**Pros:**
- Can optimize for each platform
- No runtime detection needed

**Cons:**
- Code duplication
- Hard to maintain consistency

**Not recommended for PHASE 2B.**

## Manifest Installation

### Chromium Installation

**Manual:**
```bash
# Copy native host manifest to Chromium directory
sudo cp apps/browser-extension/src/native-messaging-host.chromium.json \
  /etc/chromium/native-messaging-hosts/org.openausweis.native.json

# Ensure correct extension ID in allowed_origins
sudo sed -i 's/REPLACE_WITH_CHROMIUM_ID/YOUR_ACTUAL_ID/' \
  /etc/chromium/native-messaging-hosts/org.openausweis.native.json
```

**Automated (via .deb installer):**
```bash
# /debian/postinst
sudo cp usr/share/openausweis/native-messaging-host.chromium.json \
  /etc/chromium/native-messaging-hosts/org.openausweis.native.json
```

**Problem:** Extension ID is dynamic (different per build, test environment, release channel).

**Solution:** 
1. Store default/test IDs in package
2. Provide setup script to replace IDs
3. Document manual steps for developers

---

### Firefox Installation

**Automatic (via add-on settings):**
- Firefox automatically installs manifest when add-on is installed from AMO
- No manual installation needed

**Manual (for development):**
```bash
mkdir -p ~/.mozilla/native-messaging-hosts
cp apps/browser-extension/src/native-messaging-host.firefox.json \
  ~/.mozilla/native-messaging-hosts/org.openausweis.native.json
```

**Advantage:** Firefox uses same native host name and path; simpler deployment.

## Development Workflow

### Setup for Chromium

1. Build extension: `npm run build --workspace @openausweis/browser-extension`
2. Open `chrome://extensions`
3. Enable "Developer mode"
4. Click "Load unpacked"
5. Select `apps/browser-extension/dist` (or `apps/browser-extension/src`)
6. Copy extension ID from UI
7. Install native messaging manifest (see above)
8. Replace ID in manifest
9. Start daemon + native host

### Setup for Firefox

1. Build extension: `npm run build --workspace @openausweis/browser-extension`
2. About Firefox: `about:debugging#/runtime/this-firefox`
3. Click "Load Temporary Add-on"
4. Select `apps/browser-extension/manifest.v2.json` (or `dist` folder)
5. Add-on ID is automatically generated/fixed
6. Install native messaging manifest manually
7. Start daemon + native host

## Known Limitations

### Manifest v2 Deprecation

**Chrome 88+:** Manifest v2 is deprecated, will be removed by late 2024.

**Current status (May 2026):** Manifest v2 is deprecated in Chrome but still works.

**Firefox:** Still uses v2; no plans to remove immediately.

**Action:** 
- For PHASE 2B: Support both v2 and v3
- For PHASE 3+: Migrate both to v3 when Firefox supports it (or move to standard Web Extensions API)

### Browser-Specific Features

Some features may be browser-specific:
- Chromium: Can use Manifest v3 service workers
- Firefox: May have different permission models

**For PHASE 2B:** Stick to common API surface; avoid platform-specific features.
